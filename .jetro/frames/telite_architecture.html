⚠ Current Confusion Points
1
Docker compose shares one credential path. TELITE_POSTGRES_DB: ${POSTGRES_DB} means the backend container gets moodle as its database name — the same DB Moodle uses. Backend tables and mdl_* tables coexist.
2
Root .env has conflicting credential sets. POSTGRES_USER=moodleuser but TELITE_POSTGRES_USER=postgres. The compose file ignores TELITE_* and uses POSTGRES_* for the backend.
3
No init script for second DB/user. PostgreSQL container only creates the database from POSTGRES_DB on first init. There is no /docker-entrypoint-initdb.d/ script to create telite_backend DB or telite_backend_user.
4
Backend .env previously tried separate Postgres creds (postgres:postgres123@localhost:55432/telite_backend_db) but the server rejected them — that DB/user didn't exist. Now reverted to SQLite.
✓ Backend currently uses SQLite
The local backend runs independently on SQLite. No Postgres dependency for local dev. This is the simplest working mode right now.
ℹ Backend code already supports separation
store.py has a full DSN fallback chain: TELITE_DATABASE_URL → TELITE_POSTGRES_* → MOODLE_DB_* → POSTGRES_*. No code changes needed.
⚠ Moodle still expects PostgreSQL
docker-config.php reads MOODLE_DB_* vars. Moodle is always Postgres-backed. It cannot use SQLite.
⚠ Docker hybrid not implemented yet
The remaining work is Docker/env cleanup: init script, compose rewiring, and proper credential separation. Not app logic.
