⚙ Architecture Rules
1. Strict DB Isolation: Users cannot access or query tables in the other database.

2. Decoupled Environments: Moodle never reads TELITE_*, and the FastAPI backend never depends on MOODLE_*.

3. One Server: Low resource usage by avoiding a second heavy PostgreSQL instance.
🛡 Authentication Flow
- Frontend Client obtains tokens directly from the FastAPI Auth Session layer.

- FastAPI Backend calls the Moodle API using MOODLE_TOKEN via isolated internal service host.

🛡 SECURITY BOUNDARY
Strict DB user permission boundaries are applied via Postgres grants. The user moodleuser cannot execute queries or see tables inside telite_backend database. Conversely, telite_backend_user cannot access Moodle's moodle database tables.
⚙ Local-Dev fallback SQLite Mode
When running local development outside Docker, the FastAPI backend falls back to SQLite:
- TELITE_DB_BACKEND=sqlite
- Local Database File: telite-backend/data/telite_lms.db
- Zero credentials required for database operations.
🐳 Orchestration Init Details
A Postgres setup script at docker/init/01-create-databases.sql is mounted inside the container's /docker-entrypoint-initdb.d/. On container first boot, it creates both databases and credentials automatically.
