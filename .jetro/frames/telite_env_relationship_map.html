🛡 Backend Resolution Priority (store.py)
The FastAPI store module resolved DSN using the following priority order:
1
TELITE_DATABASE_URL
Direct Postgres URI specification string. If present, it bypasses all other variables.
2
TELITE_POSTGRES_* Group
Separate backend connection credentials (TELITE_POSTGRES_DB, TELITE_POSTGRES_USER).
3
MOODLE_DB_* Fallback (Legacy)
Checks for Moodle DB variables. Left for compatibility in simple single-db setups.
4
POSTGRES_* Fallback (Legacy)
Last-resort DB variables. Resolves to standard Postgres settings.
🚨 Identified Conflicts & Issues
Duplicate Credential Mappings in Compose
The backend container was mapping TELITE_POSTGRES_DB: ${POSTGRES_DB}. Because POSTGRES_DB=moodle, the backend was forced to use the Moodle database. The separate TELITE_POSTGRES_DB variable in the root `.env` was ignored.
SQLite Fallback Behavior
If TELITE_DB_BACKEND is set to sqlite, the resolution engine bypasses the entire PostgreSQL fallback priority chain and connects to the local SQLite database path. This prevents local runs from crashing when no Postgres server is running.
